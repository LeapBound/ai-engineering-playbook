# AI 导致工程事故的案例

## 为什么（风险）
真实案例能暴露 AI 生成带来的隐藏风险，帮助团队建立可执行的预防机制。

## 不做会怎样
团队会重复踩坑，事故复盘无法沉淀为规则，风险只能靠个人经验兜底。

## 怎么做（规则/示例）
- 用统一模板记录案例：场景描述、问题根因、如何预防。
- 将预防措施转化为规则或测试门禁。

### 案例 1：架构漂移
- 场景描述：AI 在不同模块使用不同设计模式，导致边界不一致。
- 问题根因：缺少统一的架构约束与模块接口规范。
- 如何预防：建立架构约束文档，强制模块依赖与设计模式统一。

### 案例 2：隐含假设
- 场景描述：AI 假设了不存在的接口或数据结构并直接调用。
- 问题根因：需求与接口契约未结构化表达，AI 通过猜测补齐。
- 如何预防：使用接口契约 DSL 或 Schema，生成代码前强制校验。

### 案例 3：测试盲区
- 场景描述：AI 只测试了 happy path，忽略异常与边界条件。
- 问题根因：测试规则未明确，生成策略偏向最短路径。
- 如何预防：测试 Checklist 约束覆盖异常、边界与回归场景。

### 案例 4：安全漏洞
- 场景描述：AI 生成的代码存在 SQL 注入风险。
- 问题根因：缺乏安全编码规范与安全测试门禁。
- 如何预防：强制使用参数化查询，并加入安全扫描与渗透测试。
