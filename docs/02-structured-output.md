# 结构化输出方法

## 为什么（风险）
自然语言输出不可审计、不可复用、不可版本控制，难以形成团队级的工程资产。

## 不做会怎样
每次生成结果不一致、无法 CI 集成、团队无法协作，最终出现“口头规范、执行失真”。

## 怎么做（规则/示例）
### 规则
- 使用 DSL / JSON Schema / Checklist 替代自然语言。
- 输出必须可校验、可复用、可版本控制。

### 示例
- API 测试 DSL：用结构化字段描述请求、断言、依赖。
- 代码审查 Checklist：明确安全、性能、可维护性项。

### 工具链
- Schema 校验：在提交前或 CI 中验证格式正确性。
- CI 集成：将结构化输出作为门禁输入。
