# Contributing

感谢你为 AI Engineering Playbook 做贡献。

本项目的核心目标不是堆功能，而是沉淀 **可审计、可复用、可验证** 的 AI 工程治理方法。

## 你可以贡献什么

- 方法论文档改进：概念澄清、结构优化、表达精炼
- 实践案例：真实项目中的成功/失败经验与复盘
- 规则模板：Checklist、Schema、流程约束示例
- 纠错与补充：错误链接、事实性错误、术语不一致

## 贡献原则

- 优先结构化表达，避免只给抽象口号
- 每个观点尽量附带可执行建议或可验证证据
- 保持与现有章节定位一致，减少重复内容
- 小步提交：一次只解决一个明确问题

## 内容组织建议

当前文档主线在 `docs/`：

- `docs/01-role-boundary.md`：角色边界
- `docs/02-structured-output.md`：结构化输出
- `docs/03-review-inversion.md`：Review 权限反转
- `docs/04-failure-cases.md`：失败案例
- `docs/05-practice-showcase-lucy-orchestrator.md`：实践成果展示

新增内容时，建议先判断：

1. 是补充现有章节，还是需要新增章节。
2. 是否能与“规则/示例/验证”三要素对齐。
3. 是否会影响现有导航（`mkdocs.yml`）。

## 提交流程

1. Fork 并创建分支（建议使用清晰分支名）。
2. 修改文档并自检链接、标题层级、术语一致性。
3. 本地预览（可选但推荐）：

```bash
mkdocs serve
```

4. 提交 PR，说明：
   - 你解决了什么问题
   - 为什么这样改
   - 影响哪些章节

## PR 描述建议模板

```md
## 背景
说明问题与动机。

## 变更
- 变更点 1
- 变更点 2

## 验证
- 链接检查通过
- 本地预览正常（如有）
```

## 讨论与反馈

- 问题反馈与改进建议：GitHub Issues
- 方法论讨论与实践交流：GitHub Discussions

欢迎把“踩坑经验”也贡献出来。可复盘的失败案例，和成功经验一样有价值。
